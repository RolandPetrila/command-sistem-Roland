1. Backup date ÎNAINTE de Faza 17 — lipsă critică
Backup-ul pe Google Drive e abia în Faza 17 (prioritate scăzută). Dar SQLite-ul (calculator.db) va conține traduceri, facturi, clienți, TM, glosare — date cu valoare reală, încă din Fazele 9-10. Dacă SSD-ul moare sau un update Windows corupe ceva, pierzi totul. Ai nevoie de un backup minim în Wave 0 sau Wave 1 — măcar un script simplu care copiază calculator.db + migrations/ într-un folder Google Drive sincronizat local, sau un cron job cu shutil.copy2(). Nu trebuie Drive API pentru asta.
2. SQLite concurrency — WAL mode nemenționat
FastAPI e async, iar cu WebSocket + background tasks + multiple module care scriu în același SQLite, vei lovi "database is locked" destul de repede. Trebuie specificat explicit în Wave 0: PRAGMA journal_mode=WAL; + PRAGMA busy_timeout=5000; la init. Fără asta, orice operație background (indexare FTS5, OCR batch) va bloca UI-ul.
3. Contradicție Vault între cele două documente
Documentarea zice "KeePass integrat — stochează parole site-uri, conturi, API keys, note securizate" (secțiunea 4, Faza 8). Planul zice "scope redus: doar chei API aplicație" (8.5 — IMPROVEMENT-14). Deci ce e? Diferența e semnificativă: un password manager complet e câteva sesiuni, un API key store e 30 de minute. Trebuie aliniate documentele.
4. Zero strategie de testare pentru modulele noi
Fazele 0-8 au avut teste E2E. Planul de extindere — 85 de features noi — nu menționează testare nicăieri. Nici măcar un minim: teste pe pipeline-ul de traducere (care atinge 6 API-uri), teste pe generarea de facturi PDF, teste pe migrarea DB. Recomand cel puțin: un task "tests" per fază critică (9, 10, 15) și un smoke test automatizat post-Wave 0.
5. Estimare Wave 0 probabil subestimată
1-2 sesiuni pentru: git init + refactorizare modulară + module auto-discovery + sidebar dinamic + migrare DB system + activity log migration + URL fix + notificări infra. Asta e enorm. Module auto-discovery singur (router registry backend + route manifest frontend + sidebar dinamic cu categorii colapsibile) e ușor 1 sesiune. Aș estima 3-4 sesiuni realist, altfel riști să faci compromisuri la fundație — exact locul unde nu vrei compromisuri.
6. Tailscale — PC-ul oprit = totul mort, nemenționat
Dacă PC-ul e în sleep sau oprit, Command Center-ul nu există. Nicio mențiune de: auto-start la boot (uvicorn ca Windows Service sau Task Scheduler), Wake-on-LAN de pe telefon, sau măcar un "status page" simplu care-ți spune de pe telefon dacă backend-ul rulează. Un task mic în Wave 1 ar rezolva: Windows Task Scheduler la logon → pornește backend-ul automat.
7. Google OAuth "30 min" — subestimată sever
Planul zice "30 min manual, o singură dată" pentru Google Cloud Console setup. Realitatea: aplicațiile în modul "Testing" au refresh tokens care expiră după 7 zile. Ca să eviți asta, trebuie "verificare" de la Google (care poate dura săptămâni) SAU te adaugi ca test user (limitat la 100 useri, dar pentru single-user merge). Trebuie notat explicit: "app rămâne în Testing mode, Roland adăugat ca test user, re-auth la fiecare 7 zile posibil necesar."
8. Limitele free tier-urilor nedocumentate
OpenAI free tier, Groq free tier, Azure free tier — care sunt limitele concrete? Groq e marcat [PROBABIL]. Dacă Groq free tier e 14,400 req/zi (cum era) vs OpenAI free tier de 200 req/zi — asta schimbă ordinea în fallback chain. Recomand un mic tabel cu limite reale verificate, ca să nu descoperi la runtime că fallback-ul 2 e de fapt mai generos decât fallback-ul 1.
9. Faza 14 (File Manager) — risc securitate path traversal
CRUD pe filesystem via browser, chiar și pe Tailscale, necesită sandboxing strict. Un request cu ../../etc/passwd sau ../../../Windows/System32/ trebuie blocat. Planul nu menționează nicio restricție de directoare permise (whitelist de root paths). Adaugă în specificație: allowed_roots configurabil + validare pathlib.resolve() + blocarea symlink-urilor în afara sandbox-ului.
10. Lipsă UX mobil — PWA fără responsive design menționat
PWA pe Android e specificat, dar nicăieri nu apare "responsive design" sau "mobile-first layout". Module precum File Manager, Translation Editor (side-by-side), sau Facturare au UI-uri complexe care nu se traduc automat pe un ecran de 6 inch. Recomand: un task transversal "responsive breakpoints" în Wave 0 sau Wave 1, care stabilește pattern-ul (Tailwind md: / lg: breakpoints) înainte ca modulele să fie construite cu layout fix desktop.
11. API fallback chain — lipsă retry/timeout/circuit breaker
Faza 9 menționează "API chain cu fallback automat" dar fără detalii. Ce se întâmplă când DeepL dă timeout la 30 sec? Se așteaptă 30 sec × 6 provideri = 3 minute? Trebuie specificat: timeout per provider (5s-10s), retry count (1-2), circuit breaker (dacă un provider eșuează de 3 ori consecutiv, skip 5 min), și UI feedback (progress bar cu "Trying provider 2/6...").
12. Translation Memory (9.3) — design insuficient detaliat
TM e o feature complexă cu impact mare ("reduce costul 30-50%"). Dar lipsesc detalii critice: cum se segmentează textul (propoziții? paragrafe?), ce threshold de similaritate se folosește pentru FTS5 match, cum se gestionează TM-ul când se schimbă terminologia (invalidare/actualizare), și UI-ul de review (acceptă/respinge match-uri). Merită un mini-design document separat.
